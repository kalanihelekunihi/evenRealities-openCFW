# iOS / Firmware Gap Report

Current `EvenG2Shortcuts` maps most known G2 BLE service IDs in `Sources/EvenG2Shortcuts/ProtocolConstants.swift:108-372`, so the high-level BLE surface is broad. The main gap is no longer service discovery; it is semantic parity. Several important firmware interfaces are still only heuristically modeled, partially decoded, or represented with stale assumptions.

## 1. BLE-Exposed vs Internal-Only Firmware Surfaces

### BLE-exposed surfaces

`openCFW` shows that the phone-visible API is organized around service handlers on the EUS/file channels, not around direct access to internal display code. Representative BLE-facing handlers include:

- `0x81` BoxDetect / DisplayTrigger for case state (`openCFW/src/platform/apollo510b/main_firmware/platform/service/box_detect/service_box_detect.c:2-18`). The current app now maps this as `boxDetect` / `boxDetectResponse` and reads it in `Sources/EvenG2Shortcuts/G2GlassesCaseReader.swift:51-239`.
- `0x10` Onboarding (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/pb_service_onboarding.c:2-27`). The app maps this in `Sources/EvenG2Shortcuts/ProtocolConstants.swift:151-162` and sends it in `Sources/EvenG2Shortcuts/G2OnboardingProtocol.swift:58-245`.
- `0x0D` Settings / local-data status (`docs/firmware/modules/settings-local-data-status.md`). The app maps the service broadly and now surfaces case4 tags `11/12/13/17` as a diagnostic local-data snapshot in `Sources/EvenG2Shortcuts/G2ResponseDecoder.swift`.
- `0x01` Dashboard sync (`openCFW/src/platform/apollo510b/main_firmware/app/gui/dashboard/page_state_sync.c:7-27`). The app maps the service and keeps observed state in `Sources/EvenG2Shortcuts/G2DashboardStateProtocol.swift:80-171`.
- `0xE0` EvenHub (`openCFW/src/platform/apollo510b/main_firmware/app/gui/EvenHub/evenhub_data_parser.c:2-29`). The app has a large EvenHub surface in `G2EvenHubProtocol`, `G2EvenHubSender`, `G2EvenHubResponseDecoder`, and the community app runtime.
- `0xFF` SystemMonitor (`openCFW/src/platform/apollo510b/main_firmware/app/gui/system/system_monitor.c:2-18`). The app maps this in `ProtocolConstants` and currently treats it as observed-state in `Sources/EvenG2Shortcuts/G2SystemMonitorSender.swift:3-40`.

Taken together with `ProtocolConstants.swift:281-366`, the app already exposes most known firmware-facing families: auth, dashboard/gesture, notifications/display wake, menu, translate, teleprompter, EvenAI, navigation, device/config, conversate, quicklist/health, onboarding, module configure, logger, system alert/close, box detect, ring relay, EvenHub, file transfer, and system monitor.

### Internal-only surfaces

`openCFW` also makes it clear that much of the real device behavior sits behind the BLE boundary and is not directly callable from iOS:

- Display startup/reflash routing, page-manager queuing, and inter-eye sync are internal helpers behind BLE handlers (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/protocol_common.c:5-23`, `:151-169`, `:173-252`).
- Dashboard widget serialization and LVGL page construction are internal implementation details (`openCFW/src/platform/apollo510b/main_firmware/app/gui/dashboard/dashboard.c:2105-2135`).
- EvenHub container validation, image assembly, and display startup/reflash dispatch are internal to the firmware once the BLE packet is accepted (`openCFW/src/platform/apollo510b/main_firmware/app/gui/EvenHub/evenhub_data_parser.c:12-29`, `:174-215`).
- System alert rendering and system-close dialog UI are firmware-side LVGL flows; the phone can trigger them, but it does not directly control the widget tree (`openCFW/src/platform/apollo510b/main_firmware/app/gui/SystemAlert/systemAlert.c:4-17`, `:63-95`; `openCFW/src/platform/apollo510b/main_firmware/app/gui/SystemClose/systemClose.c:4-34`).

## 2. Known Incorrect Assumptions / Behaviors in the iOS App

- Dashboard state is correctly treated as push-only at runtime, and the app now routes the real `0x01-20` packets into the dashboard parser/logging path, but the underlying serializer is still only heuristically decoded. Firmware shows dashboard sync is pushed on `0x01-20` via `send_page_state_to_app()` as page-state transitions (`enter`, `activate`, `close`, `quicklist-expanded`, `refresh`), not a stable host query/response model (`openCFW/src/platform/apollo510b/main_firmware/app/gui/dashboard/page_state_sync.c:10-27`; `Sources/EvenG2Shortcuts/G2DashboardStateProtocol.swift`; `Sources/EvenG2Shortcuts/G2ResponseDecoder.swift`).
- The app had been using an empty write on `0x0D-00` as a generic “configuration query.” That does not match the recovered settings service shape in `openCFW`, where normal phone-originated reads are protobuf commands on `0x0D-20` and `0x0D-00` is the response lane. The main config reader now uses `APP_REQUIRE_BRIGHTNESS_INFO` (`cmdId=1`) on `0x0D-20`, and the empty `0x0D-00` path should remain debug-only (`openCFW/src/platform/apollo510b/main_firmware/app/gui/setting/setting.c:459-680`; `Sources/EvenG2Shortcuts/G2ConfigurationReader.swift`).
- Several Swift-side settings helpers were over-accepting ambiguous `0x0D-00` snapshots. Firmware RE shows `0x0D` is a multiplexed envelope carrying selector updates, local-data status, and notify groups, not a clean set of dedicated per-feature response packets. The app now logs a best-effort `ConfigInterpretation` summary, rejects obvious cross-family misparses such as wear-status being interpreted as screen-off settings or vice versa, uses selector-root case3 writes for wear detection (`selector=5`) and silent mode (`selector=6`), and uses a more plausible `cmdId=2` basic-settings envelope for screen-off writes, but the exact host-side query routes for `getGlassesIsWear`, `requestScreenOffInterval`, and some “basic settings” reads are still unresolved (`docs/firmware/modules/settings-dispatch.md`; `docs/firmware/modules/settings-selector-schema.md`; `docs/firmware/modules/settings-runtime-context.md`; `Sources/EvenG2Shortcuts/G2ResponseDecoder.swift`; `Sources/EvenG2Shortcuts/G2WearDetectionProtocol.swift`; `Sources/EvenG2Shortcuts/G2GlassesSettingsProtocol.swift`; `Sources/EvenG2Shortcuts/G2UniversalSettingsReader.swift`; `Sources/EvenG2Shortcuts/G2SettingsEnvelope.swift`).
- The auth decoder had a stale pair-manager assumption: `0x80-01` `f1=6` had been treated as a generic keepalive, but `openCFW` identifies `f1=6` as `RING_CONNECT`, `f1=5` as `PIPE_ROLE`, and heartbeat as `f1=0x0E`. The app now logs these auth-side diagnostics with the firmware-backed type mapping instead of collapsing them into “keepalive” (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/pb_service_pair_mgr.c:15-31`; `Sources/EvenG2Shortcuts/G2ResponseDecoder.swift`).
- `G2OnboardingProtocol.queryHeadUpStatus()` had been modeled as an onboarding query even though the firmware evidence does not support that. The app now suppresses the invalid speculative `0x10` probe, uses raw onboarding `0x09` side-channel packets when observed, falls back to `0x0D` local-data `tag17` as a best-effort diagnostic head-up signal, and no longer treats a fake protobuf `type=4` as head-up state (`openCFW/src/platform/apollo510b/main_firmware/app/gui/onboarding/onboarding_data_manager.c:21-24`, `:107-149`; `docs/firmware/modules/settings-local-data-status.md`; `Sources/EvenG2Shortcuts/G2OnboardingProtocol.swift`).
- The onboarding decoder had stale top-level semantics. Firmware documents `0x10` protobuf message classes as `1=config`, `2=heartbeat`, `3=event`, with ACK subtypes `3/4/5` in `f3`, while `onboarding_data_manager.c` separately sends raw wear/head-up side-channel packets (`0x0D`, `0x09`) on the same service. The app now distinguishes those lanes explicitly in logs/tests, but request payload semantics remain partially heuristic (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/pb_service_onboarding.c:13-27`, `:99-107`; `openCFW/src/platform/apollo510b/main_firmware/app/gui/onboarding/onboarding_data_manager.c:21-24`, `:69-70`, `:114-117`).
- `G2SystemMonitorSender` now correctly treats SystemMonitor as observed-state, and the `SystemMonitorView` now preserves/restores the pre-existing monitor callback instead of clearing it on exit, but `G2SystemMonitorProtocol.queryPayload()` still encodes a faux query model that firmware does not support as a general status read. The firmware handler only acts on the reset magic sequence and otherwise ignores non-reset traffic (`openCFW/src/platform/apollo510b/main_firmware/app/gui/system/system_monitor.c:87-146`; `Sources/EvenG2Shortcuts/G2SystemProtocols.swift`; `Sources/EvenG2Shortcuts/SystemMonitorView.swift`).
- The app now treats any `0x21-00` data as unexpected diagnostic traffic, but the remaining parser still exists because captures may surface stale or cross-wired data. Firmware documentation for `systemAlert.c` says the service is receive-only from the phone into the glasses UI (`openCFW/src/platform/apollo510b/main_firmware/app/gui/SystemAlert/systemAlert.c:15-21`; `Sources/EvenG2Shortcuts/G2SystemProtocols.swift:3-46`).
- The app had been treating `SystemClose 0x22` as protobuf, but firmware `system_close_common_data_handler()` actually consumes a raw 3-byte request `[dialog_type, msgId_lo, msgId_hi]`, and replies likewise send raw 3-byte selection data `[selection, msgId_lo, msgId_hi]`. The Swift side now uses the raw wire format and logs the style-dependent `raw=1` ambiguity instead of pretending the selection is always self-describing (`openCFW/src/platform/apollo510b/main_firmware/app/gui/SystemClose/systemClose.c:29-33`; `:230-244`; `:971-1023`; `:1096-1118`; `Sources/EvenG2Shortcuts/G2SystemProtocols.swift`; `Sources/EvenG2Shortcuts/G2SystemCloseSender.swift`).
- `0x0A-20` had been left as an unknown “session init trigger” service. `openCFW` `service_time.c` ties it to phone time synchronization using `f128={f1=UnixEpoch,f3=TZ_offset}` where timezone is stored in 15-minute increments. The app now has a firmware-backed payload builder/probe for that lane and treats it as fire-and-forget rather than a request/response service (`openCFW/src/platform/apollo510b/main_firmware/platform/service/time/service_time.c`; `Sources/EvenG2Shortcuts/G2SessionInitProtocol.swift`; `Sources/EvenG2Shortcuts/G2DebugLogger.swift`).
- The app’s higher-level firmware log manager had still been treating a settings-lane `0x0D-20` logger probe as though it were the normal logger control path. Firmware evidence is clear that logger control belongs on `0x0F-20`, so the manager now uses the real logger service and keeps the old `0x0D-20` logger behavior only as an explicitly labeled legacy probe (`docs/firmware/modules/logger.md`; `Sources/EvenG2Shortcuts/G2LoggerProtocol.swift`; `Sources/EvenG2Shortcuts/G2LoggerSender.swift`; `Sources/EvenG2Shortcuts/G2FirmwareLogManager.swift`; `Sources/EvenG2Shortcuts/LoggerView.swift`).
- The app’s file export path had been speaking the upload/file-transfer services (`0xC4/0xC5`) with an inferred `[0x03][path]` payload. openCFW shows export is a separate service family: phase-0 export start on `0xC6`, open reply on `0xC2`, and streamed data on `0xC3`. The Swift export path now uses the dedicated export services and logs the firmware-sized open reply (file size + CRC32) instead of treating export as a variant of upload (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/efs_service/efs_service.c`; `openCFW/src/platform/apollo510b/main_firmware/comms/efs_file_service.c`; `Sources/EvenG2Shortcuts/ProtocolConstants.swift`; `Sources/EvenG2Shortcuts/G2FileExportClient.swift`; `Sources/EvenG2Shortcuts/G2ResponseDecoder.swift`).
- `0x20-00` module-config responses had been treated as bare ACKs, but firmware evidence shows at least two shapes: a minimal ACK path and richer SET-response echoes carrying command/value data. The app now logs more of this lane, but it still does not fully model the underlying schema (`openCFW/src/platform/apollo510b/main_firmware/platform/protocols/pb_service_dev_config.c:1-18`; `openCFW/src/platform/apollo510b/main_firmware/app/gui/module_configure/general_configure.c:110-173`).
- The current menu sender/parser is still more speculative than the rest of the UI stack. `openCFW` shows menu info is built around firmware `cmd_type=3` / response `cmd_type=4` with a dedicated nanopb schema, and `0x03-00` is an ACK/error envelope rather than a selected-item text response. The app now logs menu RX as `configAck/configError`, probes the real `0x03` service instead of a stale `0x0D` settings guess, and fragments long menu payloads instead of truncating them, but the per-item field mapping is still only partially recovered (`openCFW/src/platform/apollo510b/main_firmware/app/gui/menu/menu_page.c:57-99`; `:287-327`; `:337-430`; `:444-703`; `Sources/EvenG2Shortcuts/G2MenuProtocol.swift`; `Sources/EvenG2Shortcuts/G2MenuSender.swift`; `Sources/EvenG2Shortcuts/G2DebugLogger.swift`).
- Case status and dashboard state still retain legacy diagnostic fallbacks through config-oriented code paths. Those may remain useful for old captures, but they should stay explicitly diagnostic-only, not part of the production protocol model (`Sources/EvenG2Shortcuts/G2GlassesCaseReader.swift:81-103`, `:174-236`; `Sources/EvenG2Shortcuts/G2DashboardStateProtocol.swift:138-171`).

## 4. Recommended Next Priorities

1. Finish onboarding semantics next. The `0x10` protobuf-vs-raw side-channel split is now recognized, and the app now surfaces both raw onboarding head-up packets and `0x0D` local-data `tag17`, but the exact host-side head-up request route still needs to be identified from settings/local-data traffic.
2. Finish the `0x0D` settings surface. The app now classifies ambiguous `0x0D-00` packets instead of blindly parsing them, Probe/TestAll now uses `0x0D-20` for normal settings queries, and the cmdId scan ignores unrelated background traffic, but the exact selector/basic query routes for wear detection, screen-off interval, and other “basic settings” methods still need to be recovered.
3. Implement a real `0x01` dashboard decoder. The app now captures the correct lane and frames it as a page-state transition stream, but the serializer is still only heuristically modeled.
4. Continue quarantining stale protocol models. `SystemMonitor.queryPayload()` is now explicitly debug-only and the `SystemAlert` parser is only used for unexpected RX diagnostics, but both should stay out of normal feature flows.
5. Deepen typed decode for partially implemented services with strong diagnostic value: ModuleConfigure/SyncInfo, Logger live streaming/export, and Menu. Logger control is now routed through the right `0x0F` service, menu transport is now treated as ACK-driven on the real `0x03` lane, and the next step in both areas is recovering the remaining payload schemas rather than continuing to rely on stale query assumptions.
