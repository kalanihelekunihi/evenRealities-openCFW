================================================================================
INTER-MODULE COMMUNICATION PATHS
================================================================================

1. BOOT CHAIN
   Power-on → boot_cli ("boot> " prompt available)
   → lvp_core init (clocks, flash probe, LDO, stack scope)
   → uart_driver init (_uartMsgInit, VC_MESSAGE config)
   → spi_flash_driver probe (JEDEC ID, model match)
   → lvp_mode_denoise init (selected mode)
   → vad_engine init (3 params)
   → npu_engine init (wake word model load)
   → sample_app init (I2S output ready)
   → idle_mode available

2. AUDIO CAPTURE PATH
   audio_board → audio_input (LvpAudioInInit) → gsc_beamformer or imcra_noise_est
   → frame_drc → vad_engine → npu_engine → uart_driver (voice events to host)

3. AUDIO PLAYBACK PATH
   uart_driver (receive audio from host) → sample_app → gx_audio_out → i2s_driver → DAC

4. HOST COMMAND PATH (Apollo510b → GX8002B)
   uart_driver (_uartRecvCallback) → TinyFrame decode → command dispatch:
     SwitchWakeupMode → npu_engine (change wake model)
     SwitchBFMode     → lvp_mode_denoise (switch GSC/IMCRA/raw)
     ReadVersion      → lvp_core (return version string)
     QueryMicState    → audio_board (mic status)
     SetMicGain       → audio_board (PGA gain 0-48)
     I2SOutputCtrl    → i2s_driver / sample_app
     CodecMicDelay1Bit → audio_board (PDM delay calibration)
     DMICClose/Open   → audio_board (mic power control)
     GetVoiceEvent    → vad_engine + npu_engine (retrieve pending events)

5. HOST EVENT PATH (GX8002B → Apollo510b)
   vad_engine (speech onset/offset) → uart_driver (TinyFrame TX) → Apollo510b
   npu_engine (wake word detected) → uart_driver (TinyFrame TX) → Apollo510b
   Log on host: [codec.host]voice event: cmd_id=0x%04X, event_id=0x%04X

6. POWER STATE PATH
   idle_mode_init → _LvpAppSuspend + SampleAppSuspend + UartMessageAsyncSuspend
   wakeup event → _LvpAppResume + SampleAppResume + UartMessageAsyncResume
   idle_mode_exit → full pipeline restart

7. DFU PATH (from Apollo510b side)
   Apollo510b service_codec_dfu.c:
     reading boot header → parse BINH → stage1 (512 bytes) → wait "wfb"
     → stage2 (12,288 bytes, configurable baud) → wait "OK"
     → wait "boot> " prompt → firmware download → CRC32 verify
   GX8002B boot_cli handles: init, ping, write, read, flash, reboot

8. DENOISE MODE SWITCH PATH
   Host SwitchBFMode → lvp_mode_denoise:
     denoise_exit_handler (free current buffers)
     → denoise_init_handler (allocate new buffers, init selected algorithm)
     → DRC re-init (FrameDrcStateInit)
     → LvpAudioInInit (restart audio input for new mode)

9. INTER-EYE COORDINATION (via Apollo510b)
   Apollo510b manages DMIC power across both eyes:
     master: send DMIC close request to slave eye
     slave: "received request from master, closing DMIC"
     slave: "received low power response, reopening DMIC"
   The GX8002B itself is unaware of inter-eye coordination — the Apollo510b
   orchestrates DMIC power states across the two GX8002B chips (one per eye).


================================================================================
SUBSYSTEM DETAIL: LVP FRAMEWORK
================================================================================

The LVP (Low-Power Voice Preprocess) framework is NationalChip's SDK for voice
processing on GX800x series chips. Key characteristics:

  - Module lifecycle: Each module registers Suspend/Resume callbacks
  - Boot banner: Prints system info (board, version, clocks, flash, LDO)
  - Stack monitoring: Detects stack overflow at runtime
  - Memory: Custom heap with magic code validation (not libc malloc)
  - No RTOS: Bare-metal event loop with module callbacks

Log tag hierarchy (in order of abstraction):
  [LVP]              — Core system boot and initialization
  [LVP_AUD]          — Audio subsystem (VAD parameters)
  [LVP_MODE_DENOISE] — Denoise mode lifecycle and algorithm init
  [I2S_OUT_APP]      — I2S output application (SampleApp)
  [IDLE]             — IDLE power state transitions
  [AB]               — Audio Board hardware configuration
  [VC_MESSAGE]       — Voice Communication message handler
  [AOUT ERROR]       — Audio output error logging


================================================================================
SUBSYSTEM DETAIL: TINYFRAME COMMUNICATION
================================================================================

The GX8002B communicates with the Apollo510b via TinyFrame serial protocol:

  Transport: UART (uart0 or uart1)
  Protocol:  TinyFrame with CRC32 integrity
  Direction: Bidirectional (commands from host, events from codec)

Message format (from Apollo510b strings):
  cmd = 0x%04X, decomposed as NR (high byte) + TYPE (low byte)
  seq = sequence counter
  flags = message flags
  length = payload length
  crc32 = CRC32 over payload

The TinyFrame library source is at:
  D:\01_workspace\s200_ap510b_iar\third_party\TinyFrame\TinyFrame.c

Apollo510b uses separate TinyFrame instances for:
  1. Inter-eye communication (left <-> right, uart_sync.c)
  2. Codec communication (Apollo510b <-> GX8002B, drv_gx8002b.c)

Host notification mechanism:
  VC_MESSAGE notify pin: GPIO asserted by GX8002B to wake Apollo510b
  [VC_MESSAGE]NOTIFY PIN %d — pin number configuration
  [VC_MESSAGE]MASSAGE UART %d — UART port assignment
