1. FWPK CONTAINER FORMAT
================================================================================

The GX8002B firmware uses NationalChip's FWPK (Firmware Package) container
format. This is NOT the same as the EVENOTA container used by the Apollo510b.

FWPK Header (from header_hexdump.txt at offset 0x00):

  Offset  Size  Value              Field
  ------  ----  -----------------  ----------------------------------
  0x00    4     46 57 50 4B        Magic: "FWPK"
  0x04    4     00 02 00 00        Version: 0x0200 (format version 2.0)
  0x08    4     02 00 00 00        Segment count: 2
  0x0C    4     00 00 00 00        Reserved / flags
  0x10    4     01 00 00 00        Segment 0 index: 1
  0x14    4     5C 95 00 00        Segment 0 size: 0x955C (38,236 bytes)
  0x18    4     30 00 00 00        Segment 0 offset: 0x30 (48 bytes)
  0x1C    4     10 8A 7E 30        Segment 0 CRC32: 0x307E8A10
  0x20    4     02 00 00 00        Segment 1 index: 2
  0x24    4     00 4A 04 00        Segment 1 size: 0x044A00 (281,088 bytes)
  0x28    4     8C 95 00 00        Segment 1 offset: 0x958C
  0x2C    4     6C D5 81 B2        Segment 1 CRC32: 0xB281D56C

Header interpretation:
  - header_skip = 48 bytes (0x30), matching manifest.json
  - Segment 0 starts at offset 0x30, size 38,236 bytes
  - Segment 1 starts at offset 0x958C, size 281,088 bytes
  - Total: 48 + 38,236 + 281,088 = 319,372 bytes (matches binary size)
  - Each segment has an independent CRC32 for integrity verification

Post-header data (offsets 0x30-0x4F):

  0x30    4     02 80 01 01        Segment 0 header: type/flags
  0x34    4     00 00 00 00        Reserved
  0x38    4     00 00 28 00        Load address or size indicator
  0x3C    4     00 16 E3 60        Checksum or entry point
  0x40    4     00 00 6D 3C        Code size or offset
  0x44    4     00 24 D4 41        Data field (possibly entry vector)

Interrupt vector table (offsets 0x50+):
  0x50    4     00 01 00 10        Vector 0: 0x10000100
  0x54    4     30 01 00 10        Vector 1: 0x10000130
  ...repeating 0x10000130 pattern for vectors 2-31...
  0xD0    4     34 01 00 10        Vectors 32+: 0x10000134
  ...repeating 0x10000134 pattern for higher vectors...

  The vector table shows two entry points:
    0x10000100 — likely reset vector / initial entry
    0x10000130 — default handler for most interrupts (vectors 1-31)
    0x10000134 — handler for higher-numbered interrupts (vectors 32+)

================================================================================
5. DUAL-BANK BOOT SYSTEM
================================================================================

The GX8002B supports dual-bank firmware for A/B update safety:

  Bank Selection:
    "boot from first firmware" (0x0004a4ec) — Bank A selected
    "boot from second firmware" (0x0004a508) — Bank B selected

  CRC Validation:
    "Crc Error %d" (0x0004a524) — CRC mismatch on active bank

  SURG Marker:
    "SURG" found at segment 1 offset 0x0930
    Also present at segment 0 offset 0x138F
    Purpose: Likely "Staging/Upgrade" marker indicating which bank has
    the newer firmware image. Written after successful boot to confirm
    the new image is valid.

  Boot flow:
    1. Read bank selection flag from flash
    2. Load firmware from selected bank
    3. Verify CRC32
    4. If CRC fails, fall back to other bank
    5. Execute from validated bank
    6. If update pending, SURG marker indicates staging bank

6. FLASH LAYOUT (External SPI NOR)
================================================================================

The external SPI NOR flash stores firmware and configuration data.
Layout is inferred from string evidence:

  Region               Purpose                        Evidence
  -------------------  ----------------------------   --------------------------
  Bank A firmware      Primary firmware image         boot from first firmware
  Bank B firmware      Secondary firmware image       boot from second firmware
  OTP region           Factory-programmed data        read flash otp error
  Configuration        BUXX-prefixed config blocks    "BUXX" at 4 addresses
  NPU model            Wake word neural network       hey_even, hi_even labels
  Boot header          BINH + bootloader stages       BINH magic, SURG marker

Flash models and capacities:
  Vendor    Model      Capacity   String Address
  --------  ---------  ---------  ---------------
  PUYA      p25q21l    2Mbit      0x00008104
  PUYA      p25q40l    4Mbit      0x0000810c
  PUYA      p25q80l    8Mbit      0x00008114
  ESMT      en25s20a   2Mbit      0x0000811c
  ESMT      en25s40a   4Mbit      0x00008128
  ZBIT      zb25wq80a  8Mbit      0x00049d68 / 0x00013a02

The largest supported flash is 1MB (8Mbit). With a 319KB firmware binary,
dual-bank storage (~638KB) fits in the 1MB parts, leaving ~386KB for NPU
model, configuration, and OTP.

Flash probe process:
  1. SPI bus init
  2. JEDEC ID read ("Flash JEDEC = %#x")
  3. Model match against supported table
  4. Flash vendor logged ([LVP]Flash vendor:[%s]) — PUYA, ESMT, or ZBIT
  5. Flash type, ID, size logged
  6. If no match: "spi flash bus: error %d reading JEDEC ID"
  7. Probe timeout: "flash probe waitting timeout."

7. NVM / OTP REGIONS
================================================================================

OTP (One-Time Programmable):
  Evidence: "read flash otp error" (0x0004998c)
  Purpose: Factory calibration data, chip ID, or security keys
  Access: Read via SPI flash OTP command set
  Error handling: Logs error on read failure (no retry evident)

NVM Configuration Blocks (BUXX):
  The BUXX magic marker appears at 4 distinct addresses:
    0x000133e4  (within segment 0 string region)
    0x00017330  (within segment 0 near wake word labels)
    0x000433f0  (within segment 1 near version string)
    0x0004de98  (within segment 1 upper region)

  BUXX likely serves as a configuration block magic marker, similar to how
  the Apollo510b uses magic markers for NVM data validation. The multiple
  occurrences across both segments suggest it is embedded in both firmware
  banks for configuration persistence.

9. SEGMENT 1 KEY MARKERS
================================================================================

Segment 1 (BINH dual-bank image) key structural markers:

  Offset    String    Purpose
  --------  --------  ------------------------------------------
  0x0000    BINH      Boot image header magic
  0x0930    SURG      Staging/Upgrade marker for bank selection

Segment 1 contains a complete duplicate of the firmware including all string
tables, allowing the GX8002B to boot from either bank independently. The
BINH header provides the boot protocol parameters (stage sizes, baud rates)
that the Apollo510b DFU driver needs to load the codec firmware.
