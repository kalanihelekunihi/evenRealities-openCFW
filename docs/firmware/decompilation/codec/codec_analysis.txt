4. COMMUNICATION PROTOCOL DETAILS
================================================================================

4.1  GX8002B ↔ Apollo510b Communication
----------------------------------------

Transport: UART serial (uart0 / uart1)
Protocol:  TinyFrame with CRC32

Message format (from main firmware strings):
  Pack message: cmd=0x%04X(NR=0x%02X, TYPE=0x%02X),
                seq=0x%02X, flags=0x%02X, length=%d, crc32=0x%08X

  NR   = message number / command class
  TYPE = sub-command type
  seq  = sequence counter
  flags= message flags
  CRC  = CRC32 (standard zlib, NOT CRC16)

4.5  DFU (Device Firmware Update) Protocol
------------------------------------------
From main firmware [codec.dfu] strings:

Boot sequence:
  1. reading boot header ...
  2. start boot stage1 ... → send 512 bytes
  3. download boot stage1 ... → wait for "wfb" prompt
  4. boot stage1 ok !
  5. start boot stage2 ... → send 12,288 bytes
  6. download boot stage2 ... → wait for "OK" prompt
  7. boot stage2 ok !
  8. waiting "boot>" ... → wait for boot prompt
  9. get "boot>" ! → boot complete

BINH header fields:
  +0x00: "BINH" magic
  +0x04: 0x55AA55AA sync marker
  +0x08: stage1_size (512 bytes)
  +0x0C: stage2_baud_rate
  +0x10: stage2_size (12,288 bytes)
  +0x14: stage2_checksum (0x20000000)
  +0x18: entry vector word
  +0x1C: handler word

DFU validation:

  [codec.dfu]FW firmware CRC32 verification failed
  [codec.dfu]Codec version is different from package version, starting upgrade
  [codec.dfu]Codec version is same as package version, no upgrade needed

The Apollo510b manages the I2S clock/data lines to the GX8002B. Audio data
flows bidirectionally:
  - GX8002B → Apollo510b: Processed mic audio (for LC3 encoding + BLE TX)
  - Apollo510b → GX8002B: Audio playback (for DAC output to speaker)

8.4  TinyFrame Protocol Details
-------------------------------
The TinyFrame serial protocol is used for BOTH:
  1. Inter-eye communication (left ↔ right glasses, uart_sync.c)
  2. Codec communication (Apollo510b ↔ GX8002B, drv_gx8002b.c)

These are separate UART links with independent TinyFrame instances.

Codec TinyFrame messages use CRC32 (unlike the BLE protocol which uses CRC16).

8.5  Audio Pipeline in G2 Architecture Context
-----------------------------------------------

  Physical Mic → GX8002B DMIC → PGA → PDM→PCM → Beamforming/IMCRA →
  DRC → VAD → NPU (wake word) → I2S → Apollo510b → LC3 Encode → BLE TX

  BLE RX → Apollo510b → LC3 Decode → I2S → GX8002B → DAC → Speaker

The GX8002B does NOT perform LC3 encoding/decoding — that is handled by the
Apollo510b's audio service layer. The codec chip provides:
  - Raw audio capture with hardware preprocessing
  - Wake word detection (always-on, low-power NPU)
  - Noise reduction and beamforming
  - Audio output to speaker via DAC
