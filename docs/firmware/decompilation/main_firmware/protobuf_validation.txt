1. AUTH (Service 0x80)
   Firmware schemas: AuthResponse (DAT_0049eaac), AuthKeepalive (DAT_0049f408)
   SDK files: ProtocolConstants.swift, G2ResponseDecoder.swift
================================================================================

FIRMWARE TX SCHEMA (AuthResponse):
  f1: uint32 type          (4=auth_success, 5=auth_response)     [HIGH]
  f2: uint32 msg_id        (connection handle / eye ID)           [HIGH]
  f3: uint32 conn_handle   (u16 at offset 2)                     [MEDIUM]
  f4: bool   status        (param_1 != 0)                        [MEDIUM]
  f5: uint32 padding       (always 0)                            [LOW]

FIRMWARE TX SCHEMA (AuthKeepalive):
  f1: uint32 type          (6=keepalive)                         [HIGH]
  f2: uint32 msg_id                                              [HIGH]
  f3: uint32 auth_flag     (auth mode byte)                      [MEDIUM]
  f4: message time_data    (system_time_get)                     [MEDIUM]
  f5: uint32 tz_param      (timezone byte)                       [MEDIUM]
  f6: uint32 end_marker    (always 0)                            [LOW]

SDK RX HANDLING (G2ResponseDecoder.swift):
  0x80-00: Heartbeat echo (type=14) parsed into G2HeartbeatEcho
           Non-heartbeat treated as auth success indicator
  0x80-01: Auth success (type=4, f3={f1=1}) or keepalive (type=6)
  0x80-02: Transport ACK (8-byte header-only)

COMPARISON:
  Field Match:  f1, f2 match (type, msg_id)

  GAPS FOUND:
  [GAP-AUTH-1] AuthResponse f3 (conn_handle) NOT parsed by SDK
     - Firmware sends connection handle in f3 of auth response
     - SDK only checks f3 for nested {f1=1} success flag
     - Potential: conn_handle could be used for eye identification

  [GAP-AUTH-2] AuthResponse f4 (status bool) NOT parsed by SDK
     - Firmware encodes a boolean status field
     - SDK does not extract this, only reads nested f3.f1

  [GAP-AUTH-3] AuthKeepalive f3-f6 NOT parsed by SDK
     - auth_flag, time_data, tz_param are sent by firmware in keepalive
     - SDK logs "Auth keepalive received (type=6 on 0x80-01)" but discards fields
     - Potential: time_data contains system clock, useful for time sync verification

  [GAP-AUTH-4] Heartbeat echo f3/f4/f5 speculatively mapped but never confirmed
     - G2HeartbeatEcho has isWearing, isCharging, isInCase (all nil)
     - Firmware heartbeat handler (PB_RxBaseConnHeartBeat) validates 0xAA magic
     - These fields may not exist in heartbeat echoes at all

  MISMATCHES: None (field numbers align)
  TYPE COMPATIBILITY: OK (varint maps to UInt8/UInt64 in SDK)

3. EVEN AI (Service 0x07)

FIRMWARE TX SCHEMAS:
  EvenAIBasic (types 1-4): f1=type, f2=msgId, f3=command, f4=flags
  EvenAIExtended (types 3,5): f1=type, f2=msgId, f3=command, f4=payload, f5=term
                               SKILL: f6, f7 additional
  EvenAIAdvanced (types 7-10): f1=type, f2=msgId, f3=command, f4=flags, f5-f9 payload
                                HB: flags=0(codec ready)/8(not ready), field_count=11
  EvenAIConfig (type 10): f1=type, f2=msgId, f3-f5=config, f6-f13=additional

SDK TX (G2EvenAIProtocol.swift):
  CTRL enter/exit:  f1=1, f2=magic, f3=nested{f1=enter/exit}
  ASK:              f1=3, f2=magic, f5=nested{textInfo}
  REPLY:            f1=5, f2=magic, f7=nested{textInfo}
  SKILL:            f1=6, f2=magic, f6=nested{f1=skillId, f2=value}
  ANALYSE:          f1=4, f2=magic, f5=nested{textInfo}
  PROMPT:           f1=7, f2=magic, f5=nested{textInfo}
  EVENT:            f1=8, f2=magic
  CONFIG:           f1=10, f2=magic, f3=nested{f1=streamSpeed, f2=textMode}
  HEARTBEAT:        f1=9, f2=magic

SDK RX (G2ResponseDecoder.swift):
  0x07-00: f1=161 → COMM_RSP (completionType from f12.f1)
           f1=2 → VAD (isActive from f3.f1)
           f1=1/3/5/6 → EvenAI echo
           else → EvenHub decoder
  0x07-01: f1=161 → completion
           else → status (f3.f1=mode)

COMPARISON:
  Field numbers: MATCH for all command types

  [GAP-AI-1] HEARTBEAT firmware checks codec state (flags=0/8)
     - Firmware sets puVar1[5] = 0 if codec ready, else 8
     - SDK heartbeatPayload() sends only f1=9, f2=magic — NO codec state field
     - SDK is TX (phone→glasses) so firmware codec state only matters for RX direction
     - But the firmware HEARTBEAT response field_count=11 means 11 fields are encoded
     - SDK only decodes f1/f2 of heartbeat responses — missing fields 3-11

  [GAP-AI-2] CONFIG firmware encodes 13 fields (field_count=0x0D)
     - Firmware: puVar1[4]=*param_2, puVar1[5]=param_2[1], puVar1[6]=0, plus f6-f13
     - SDK configPayload sends: f1=10, f2=magic, f3=nested{f1=speed, f2=mode}
     - Only 3 fields sent vs firmware's 13 possible — 10 config fields unused
     - Potential: Additional config for AI persona, language, context window, etc.

  [GAP-AI-3] SKILL firmware uses field_count=7 and separate buffer DAT_004efc7c
     - Firmware encodes 7 fields for SKILL; SDK sends f1, f2, f6 (3 fields)
     - Fields 3-5 and f7 are not populated by SDK
     - Potential: f3=command byte, f4/f5=payload, f7=additional params

  [GAP-AI-4] PROMPT firmware uses field_count=9 on EvenAIAdvanced table
     - SDK sends: f1=7, f2=magic, f5=textInfo (3 fields)
     - Firmware encodes up to 9 fields
     - Missing fields 3, 4, 6-9 could carry prompt parameters (temperature, max_tokens, etc.)

  [GAP-AI-5] EVENT firmware uses field_count=10 with auto-increment at DAT_004f05c8
     - SDK eventPayload sends: f1=8, f2=magic only (2 fields)
     - Firmware encodes 10 fields — 8 fields undiscovered
     - EVENT has auto-increment counter separate from phone msgId

  [GAP-AI-6] EvenAI echo for ANALYSE (type=4) not handled
     - SDK handles echoes for f1=1/3/5/6 but NOT f1=4 (ANALYSE)
     - Firmware's type=2 encoder is labeled ANALYSE echo (field_count=4)
     - When glasses echo an ANALYSE command, SDK falls through to EvenHub decoder

  [GAP-AI-7] REPLY via task queue — firmware enqueues type=5 as task_queue(5, data, 0x208)
     - SDK sends REPLY synchronously, firmware processes async
     - No impact on wire format but explains why REPLY responses may be delayed

  MISMATCHES:
  [MISMATCH-AI-1] SDK ASK uses f5 for text, firmware EvenAIBasic uses f3 for command
     - SDK: f1=3, f2=magic, f5=textInfo
     - Firmware basic: f1=3, f2=msgId, f3=command, f4=flags
     - Firmware ASK large: f1=3, f2=msgId, f3=command, f4=payload, f5=terminator
     - SDK puts text in f5 which maps to firmware's f5 (terminator) in basic table
     - BUT: firmware ASK large table has f4=bytes payload for large content
     - The SDK format works because the large table's f4 accepts the text payload
     - Still, the SDK skips f3 (command byte) which firmware expects
     - RISK: Medium — firmware may use f3 to determine ASK sub-type

  TYPE COMPATIBILITY: OK (varint→UInt8/UInt64, bytes→Data)
