Notes on task creation:

  - Priority 0 = osPriorityNormal (idle-level for CMSIS)
  - Priority 1 = osPriorityAboveNormal
  - Despite the names "agingTimer" and "powerOnTimer", both are created via
    osThreadNew (FUN_0800a860), making them threads, not software timers.
    They likely block on events and fire callbacks periodically.

TASK DETAILS:

  defaultTask (priority 0):
    Entry: FUN_08007038 (defaultTask_body)
    Main supervisory loop, runs every 1000ms (osDelay(1000) at bottom).
    Recursively calls itself from the bottom of the loop.
    Responsibilities:
      - Counter modulo 5 for periodic polling
      - Glasses detection state monitoring
      - Charging state management (L/R per-side)
      - Water detection timeout (300s fake standby counter)
      - OTA firmware check trigger (after 20+ iterations, bat>50%, bat>20%)
      - Idle mode entry (30 iterations with no glasses, no USB)
      - Standby entry (idle, gls_bat_full, low_bat)
      - Event flag polling: bit 1 = timer reschedule, bit 12 = exit idle mode
      - IWDG refresh before standby
      - Sleep mode entry when idle counter > 0x1D

  glsDetectTask (priority 1):
    Entry: FUN_0800cc4c (glasses_detect_ir_scan)
    Performs IR LED scan to detect glasses in/out per side.
    Algorithm:
      1. Drive IR LED pin (GPIO 3=0xAF/4=0xAF for L, 4=0xAF/3=0xAF for R)
      2. Wait 3ms (osDelay)
      3. Read ADC (adc_read_ir_sensor @ 0x08002e94)
      4. Threshold: ADC*1000/ref < 0x50 = glasses present
      5. ADC*1000/ref >= 0x79 = definitely absent
      6. Water detection: 6 consecutive "present but leaking" = disable 5V
    Signals defaultTask via LED timer restart and state struct updates.
    Also performs secondary IR verification scans.

  pwrManagerTask (priority 1):
    Entry: FUN_080082c6 (pwrManagerTask_stub) — EMPTY BODY (just returns)
    Likely placeholder for future power management features.
    The actual power management logic is in defaultTask.

  ledTask (priority 1):
    Controls LED indicators for charging status, error conditions, aging.
    LED states:
      - phase 0: idle (cleared on GLS status change)
      - phase 1: charging in progress
      - phase 2: cleared/transition
    LED patterns driven by gpio_gls_l_charge_start / gpio_gls_r_charge_start.

  agingTimer (thread, priority 0):
    Aging test coordinator — manages the factory aging test state machine.
    Works with aging_timer_callback (0x08002c70) for periodic checks.

  powerOnTimer (thread, priority 0):
    Manages power-on LED sequence after boot.

  FreeRTOS Idle Task (internal, priority 0):
    Entry: freertos_idle_task (0x0800c864)
    Calls freertos_idle_hook_stub (0x0800c884) — empty hook.
    Standard FreeRTOS idle behavior.

  Timer Daemon Task (internal):
    Entry: freertos_timer_daemon_task (0x0800c2f6)
    Processes software timer commands from the timer queue.

FREERTOS SOFTWARE TIMERS (4 total)
================================================================================

All timers created via osTimerNew wrapper (FUN_0800a754), which calls
xTimerCreate (FUN_0800b18c) internally.

#  Name                 Params Offset  Handle Offset  Type          Notes
-- -------------------  -------------  -------------  ----          -----
1  clearLedTimer        +0x84          +0x2C          One-shot      Clears LED after delay
2  showErrorLedTimer    +0xA8          +0x30          One-shot      Shows error LED pattern
3  startLedTimer        +0xCC          +0x34          One-shot      Starts LED sequence
4  setAgingStatusTimer  +0x60          +0x28          One-shot      Sets aging status on YHM2510

Timer creation: osTimerNew(name, type=0, params_ptr)
  - type=0 in CMSIS-RTOS2 = osTimerOnce (one-shot)
  - All 4 timers use type=0 (second parameter to FUN_0800a754)

Timer periods are set dynamically via osTimerStart (FUN_0800a6a0) or
osTimerChangePeriod (FUN_0800a908). Observed periods (in ms):

  Timer                  Period (ms)  Context
  ---------------------  -----------  -------
  clearLedTimer          4            Aging status check
  showErrorLedTimer      4            Aging status check
  startLedTimer          1000         restart_led_timer_1s (0x0800b98c)
  setAgingStatusTimer    128 (0x80)   Exit aging status
  Various contexts:      64 (0x40)    Serial debug timer
                         256 (0x100)  YHM2510 dump (0xB0 debug cmd)
                         8            UART frame timeout
                         1024 (0x400) Glasses command relay

Timer operations observed:
  osTimerStart  (FUN_0800a6a0): Start timer with specified period
  osTimerStop   (FUN_0800a818 / FUN_0800a93e): Stop running timer
  osTimerChange (FUN_0800a908): Change period of running timer

Timer callback:
  aging_timer_callback (0x08002c70): Called from defaultTask on counter%5==0
  when aging mode is active. Not a true FreeRTOS timer callback — it's polled.

restart_led_timer_1s (0x0800b98c):
  Stops the defaultTask timer, then restarts it with 1000ms period.
  Called when glasses detection state changes to trigger LED update.

================================================================================
FREERTOS EVENT FLAGS (via appEvent queue/event group)
================================================================================

The appEvent handle at offset +0x38 is used with osEventFlagsWait/Set:

  osEventFlagsWait (FUN_0800a62e): defaultTask polls for event bits
  osEventFlagsSet  (FUN_0800a5ea): ISRs/timers signal events

Observed event flag bits:
  Bit 1  (0x02):    Timer reschedule — triggers osTimerChangePeriod on defaultTask timer
  Bit 12 (0x1000):  Exit idle mode — triggers exit_idle_mode (FUN_0800137c)

Event flow:
  1. ISR (EXTI, UART) or timer callback sets event flag
  2. defaultTask polls appEvent with osEventFlagsWait
  3. On bit match, clears the bit and executes corresponding action

FREERTOS SEMAPHORES / MUTEXES
================================================================================

FreeRTOS kernel includes:
  xSemaphoreCreateMutex  (FUN_0800b2bc) — present in kernel
  xSemaphoreCreateBinary (FUN_0800b2d8) — used by osThreadNew for task creation

No application-level mutexes or semaphores were identified in the box firmware.
The binary semaphore is used internally by the RTOS for task notification
(ulTaskNotifyTake / xTaskNotifyGive pattern).

Task synchronization is primarily done through:
  - Event flags (appEvent group)
  - Shared state structs (accessed by multiple tasks)
  - Critical sections (enter_critical/exit_critical at 0x080000f4/0x080000fc)
